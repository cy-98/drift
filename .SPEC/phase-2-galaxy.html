Phase 2.5 星系

目标：在星区之上增加宏观气候带——星系不是关卡，而是同一套「宇宙天气」覆盖的星区群；可落在 data/galaxies/*.yml，后续可调物理法则与丰富内容。

0/14 · 0%

已确认的设计决策

  • 尺度design

    1 星系 = 4×4 星区 ≈ 2880×2880 世界单位。

  • 原型数量design

    4 种：静湖 / 余炌 / 薄雾 / 空廊。

  • 星门语义wormhole

    区内折跃保留；稀有隑间门跨相邻星系。

  • 实现优先级roadmap

    G0 → G1 先行；G2–G4 后续。

  • 定义文件data/galaxies/

    YAML/JSON 描述气候与内容权重；physics 字段预留。

四种原型（Archetype）

静湖

still-lake

偏冷蓝、湖面光强；POI 偏冻结/湖类。

余炌

ember

暖色星云、星尘略密；稀有 POI 权重略高。

薄雾

mist

雾浓、星点柔；遗迹略多、POI 略稀。

空廊

void-hall

极稀疏、背景更深；适合久漂、几乎无压力。

界面位置显示（HUD）

HUD = Head-Up Display，即演玩时屏幕上的轻量信息层（速度、高度、当前星区名等）。计划在现有星区行上方增加星系行：星系 · 薄雾弧群 / 星区 · 静潮星域。

G0 命名层（优先）

  • 星系坐标与命名galaxies.js

    还差什么：galaxyCoords / galaxyLabel / galaxyMeta；与 sectors.js 同样纯函数。

  • HUD 双行位置dom-hud.js

    还差什么：星系名 + 星区名；进入新星系时更新。

  • 进入星系 lorejournal.js

    还差什么：跨星系边界字幕 + 日志自动记录；可关闭。

  • 星系定义文件data/galaxies/

    还差什么：加载 *.yml / JSON；无文件时程序化 fallback；示例见 _schema.example.yml。

G1 气候层（优先）

  • 四原型 paletteworld.js

    还差什么：星云/雾/星密/湖光由星系盘驱动；2–3s 插值过渡。

  • POI 权重pois.js

    还差什么：pickType 参考 galaxyMeta.poiWeights。

  • 渲染合成world.js

    还差什么：galaxy × sector × 日暮 三层相乘。

G2 交通层（后续）

  • 区内星门语义wormhole.js

    还差什么：明确同星系内大距离折跃；文案区分。

  • 隑间星门wormhole.js

    还差什么：稀有生成；跳至相邻 (gx±1,gz±1) 种子位置。

  • 造访星系档案journal.js

    还差什么：去重列表、随档案导出。

G3 记忆层（后续）

  • 星标带 galaxyIdbookmarks.js

    还差什么：档案与设置列表显示所属星系。

  • 星座按星系归档constellation

    还差什么：同星系解锁连线可略亮。

  • 星系成就achievements.js

    还差什么：如「阅尽四象」、「隑门旅人」。

G4 星图 UI（可选）

  • 设置页抽象星图settings

    还差什么：2D 点阵显示已造访 gx,gz；非必须。

明确不做

无边界加载、无必须清图；全部 procedural + 可选单格 yml 覆盖。

Spec 预留 physics 字段；首批 G0/G1 不改飞行手感。

概念设计见 概念设计；星区已完成项见 Phase 2。
